# 바이브 코딩에서 PRD와 디자인의 역할

## PRD가 소용없어진 게 아니다

배너 생성기를 만들 때 PRD를 먼저 썼다. 그 덕분에 v2.0에서 타입 구조를 한번에 깔끔하게 잡을 수 있었다. 배너 4종, 키워드 체계, 이미지/그래픽 모드 구분 — 이런 뼈대가 PRD에서 나왔다.

문제는 PRD를 한번 쓰고 버렸다는 거다. v2.0 이후로 PRD를 안 봤다. 팔레트 칩이 생기고, 인스턴스 복제가 추가되고, 이미지형이 대폭 바뀌었는데 PRD에는 반영 안 됐다.

그렇다고 PRD를 계속 업데이트할 필요는 없다. 바이브 코딩에서는 코드가 곧 스펙이다. PRD를 살아있는 문서로 관리하는 건 오히려 시간 낭비다.

## PRD는 "출발점"으로만 쓴다

전통적인 PRD는 계약서다. "여기 적힌 거 다 만들어야 함." 바이브 코딩에서 PRD는 출발점이다. "여기서 시작해서 만들면서 바꿀 거임."

### PRD에 들어갈 것

처음에 한번만 쓴다. 짧게.

```markdown
## 배너 생성기
- 목적: 비디자이너가 배너를 직접 만들 수 있는 웹 툴
- 배너 종류: 롤링, BM, 팝업, 콘텐츠
- 모드: 그래픽형, 이미지형
- 핵심 기능: 키워드 선택 → 자동 테마 생성 → 세부 조정 → 내보내기
```

AI가 이걸 보고 타입 구조를 잡는다. `BannerType = 'rolling' | 'bm' | 'popup' | 'content'`가 한번에 나온다. 이게 PRD의 역할이다.

### PRD에 안 들어갈 것

- 상세 UI 명세 (만들면서 바뀔 거니까)
- 기능 목록 전체 (써보면서 추가/삭제될 거니까)
- 우선순위 (그때그때 판단할 거니까)

## 기능 먼저, 디자인 나중에

배너 생성기를 만든 순서:

```
PRD → MVP(기능 위주) → 기능 수정/구조 변경 → 디자인(사이트 UI)은 아직 안 함
```

이 순서가 맞다. 이유는 단순하다.

### 기능이 흔들리는데 디자인하면 버린다

04-10 세션에서만 해도:
- 팔레트 칩 6개가 그라데이션 섹션 안에 새로 생겼다
- 그래픽 에셋 선택기가 이미지형에 추가됐다
- 세부설정에서 그래픽 섹션이 통째로 사라졌다
- 배경 꾸밈 드롭다운에서 '없음' 옵션이 빠졌다

이걸 미리 디자인했으면 4번 다시 그려야 했다. 기능이 바뀌는 중에 디자인 에셋을 만드는 건 낭비다.

### 디자인은 기능이 안정된 후에

기능이 더 이상 크게 안 바뀔 때 디자인을 시작한다. 이걸 판단하는 기준:

- 새 기능 추가보다 기존 기능 다듬기가 많아졌다
- "이것도 넣자"보다 "이거 좀 불편한데"가 많아졌다
- 구조(타입, 상태 관리)를 바꾸지 않고 UI만 고치는 단계다

배너 생성기는 지금 이 단계에 와 있다. 인스턴스 시스템, 이미지형 개선, 에셋 카테고리 분류가 다 들어갔다. 이제 디자인을 할 차례다.

### 디자인 시작하는 방법

전체를 새로 디자인하지 않는다. 이미 동작하는 화면이 있으니까.

**1단계: 현재 화면을 스크린샷 찍는다**

지금 상태가 곧 와이어프레임이다. 레이아웃, 정보 구조, 인터랙션 흐름이 이미 잡혀 있다.

**2단계: 불편한 부분만 표시한다**

"여기 버튼이 너무 작다", "여기 간격이 이상하다", "이 드롭다운은 칩으로 바꾸고 싶다" — 전체 리디자인이 아니라 문제 기반 개선.

**3단계: 컴포넌트 단위로 다듬는다**

한번에 전체를 바꾸지 않는다. InputPanel 먼저, 그다음 DetailSettingsPanel, 그다음 BannerPreviewCard. 하나 다듬고 적용하고, 다음으로 넘어간다.

이것도 바이브 코딩이다. 디자인을 피그마에서 완성한 다음 한꺼번에 적용하는 게 아니라, 하나씩 고치면서 확인한다.

## 전체 흐름 요약

```
[시작]
  PRD (짧게, 뼈대만)
    ↓
[기능 단계]
  MVP → 써보기 → 고치기 → 반복
  PRD는 안 본다. 코드가 스펙이다.
    ↓
[안정 단계]  ← 지금 여기
  기능 추가가 줄고, "불편하다"가 늘어남
    ↓
[디자인 단계]
  스크린샷 기반 → 문제 표시 → 컴포넌트 단위 개선
  피그마 완성본을 안 만들고, 바이브로 다듬는다
```
