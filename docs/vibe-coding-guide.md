# 바이브 코딩 가이드

## 바이브 코딩이 뭔가

화면을 보면서 "이거 이상한데" → AI한테 말하기 → 고쳐짐 → 다른 게 보임 → 또 말하기. 기획서 없이, 만들면서 발견하고, 발견하면서 고치는 방식이다.

이게 잘 돌아가는 이유는 **실물을 보면서 판단**하기 때문이다. 기획서로 "이미지형 배너의 배경색을 이미지의 주요 색상에서 자동 추출하여..." 라고 쓰는 것보다, 실제 화면에서 "배경이 항상 검정이네, 사진 색이랑 맞추면 안 돼?" 하는 게 더 정확하다.

## 흐름을 끊지 마라

콘페티 색상 바꾸다가 보더가 보여서 고치고, 고치다 보니 배경색이 고정이라 추출하고, 추출하니 팔레트가 필요하고, 팔레트 넣으니 텍스트가 안 보여서 대비를 넣고 — 이게 자연스러운 흐름이다.

"오늘은 여기까지만" 하고 자르면 맥이 끊긴다. 흐름을 따라가는 게 바이브 코딩의 핵심이다. 범위를 제한할 필요 없다.

## 대신 체크포인트를 찍어라

흐름을 끊지 않되, 하나 만들 때마다 **10초만** 쓴다.

### 체크 1: 다른 케이스로 해봐

기능이 동작하면, 다른 조건으로 한번만 더 해본다.

```
배경색 추출 됐다 → 밝은 사진으로도 해보자 → 어두운 사진으로도 → 무채색으로도
텍스트 대비 됐다 → 분홍 배경에서도 되나? → 검정 배경에서도?
```

이 세션에서 HSL 버그(분홍 배경에 녹색 텍스트)를 발견한 건 분홍 사진을 넣어봤기 때문이다. 안 넣었으면 그냥 넘어갔을 거다.

### 체크 2: 전부 적용됐나 봐

같은 패턴이 여러 곳에 있으면, 전부 바뀌었는지 확인한다.

```
배경색을 바꿨다 → 롤링, BM, 팝업, 콘텐츠 4개 다 됐나?
```

이 세션에서 롤링/BM만 고치고 팝업/콘텐츠를 빠뜨렸다. "4개 배너 타입이 있다"는 걸 알면 2초면 확인할 수 있는 건데 안 했다.

### 체크 3: AI가 숫자 말하면 의심해

AI가 "0개", "없음", "전부" 같은 절대적 표현을 쓰면 한번 더 물어본다.

```
❌ "growth 에셋이 0개입니다" → "아 부족하네"
✅ "growth 에셋이 0개입니다" → "진짜? 실제 폴더에서 세봐"
```

이 세션에서 에셋 커버리지 분석이 완전히 틀렸다. 코드에서 빈 배열만 보고 "없다"고 했는데, 실제로는 동적 로딩으로 16개가 있었다. "어딜 보고 분석한 거야?" 한마디가 2시간 낭비를 막았다.

## 끝날 때 1분

세션 끝에 세 줄만 말하면 된다.

```
한 거: 이미지형 배경 추출, 팔레트, 텍스트 대비
실수한 거: HSL 버그, 팝업 빠뜨림
다음에: 블러 내보내기 확인
```

AI가 메모리에 저장한다. 다음 세션에서 불러온다.

## 정리

| | 하는 것 | 시간 |
|---|---|---|
| **만들기** | 흐름대로. 끊지 말고 따라간다 | 자유 |
| **체크 1** | 다른 케이스 하나만 더 | 10초 |
| **체크 2** | 같은 패턴 전부 됐나 | 10초 |
| **체크 3** | AI 숫자/판단 의심 | 10초 |
| **정리** | 한 거 / 실수 / 다음 | 1분 |

바이브 코딩의 속도를 유지하면서, 확인 습관만 넣으면 퀄리티가 올라간다. 느려지는 게 아니라, 나중에 돌아와서 고치는 시간이 없어지는 거다.
