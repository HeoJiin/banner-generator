## v2.0 — 기반을 새로 깔다 (04-08)

### 뭘 했나
키워드 체계를 완전히 갈아엎었다. 이전에는 "할인/신규/감사" 3개였는데, "혜택/정보 제공/성장/긴급" 4개로 바꿨다. 소구점이 더 명확해졌다.

가장 큰 변화는 **동적 에셋 로딩**이다. 350개 넘는 3D 에셋을 일일이 코드에 등록하는 대신, 파일 이름 규칙(`benefit-`, `info-`, `urgent-` 등)으로 자동 분류하게 만들었다. 에셋 파일만 폴더에 넣으면 코드 수정 없이 바로 반영된다.

저장/불러오기도 이때 들어갔다. localStorage에 상태를 통째로 저장하고, 나중에 불러올 수 있다.

### 왜 이렇게 했을까
에셋이 수백 개인데 하나하나 코드에 쓸 수 없었을 것이다. 파일명 기반 자동 매핑은 꽤 영리한 선택이다. 다만 파일명을 잘못 지으면 분류가 안 되는 위험이 있다.

### 이때의 기술 결정
- 타입 정의를 먼저 꼼꼼하게 짰다 (`banner.ts` 738줄)
- 상태 관리를 zustand 같은 라이브러리 없이 `useState` + `useCallback`으로 했다
- 배경 꾸밈에서 동전(money) 에셋을 버렸다 (무겁고 트렌드에 안 맞아서)

## v3.0 — 디테일을 다듬다 (04-08)

### 뭘 했나
배너마다 타이틀을 다르게 쓸 수 있게 됐다. 전역으로 "봄 신상 입고!" 쓰되, BM 배너만 "봄 맞이 할인"으로 바꿀 수 있다.

그라데이션을 3색(시작-중간-끝)에서 2색(시작-끝)으로 줄였다. 중간색이 있으면 조합이 복잡해지는데, 실제로 쓸 때는 2색으로 충분했던 것 같다.

"다시 뽑기"도 분리했다. 이전에는 다시 뽑으면 배경색+에셋+꾸밈이 전부 바뀌었는데, 이제 시즌 배경만 따로 다시 뽑을 수 있다.

### 왜 이렇게 했을까
v2.0을 실제로 써보니까 불편한 점이 보였을 것이다. "배너마다 텍스트가 달라야 하는데", "다시 뽑으면 마음에 든 에셋까지 바뀌어버려" 같은 피드백.

### 주목할 점
- 이전 버전에서 저장한 데이터를 불러올 때 새 필드가 없어도 깨지지 않게 **패칭 로직**을 넣었다
- 3색 그라데이션을 2색으로 줄인 건, 단순화가 오히려 사용성을 올린 케이스다

## v4.0 — 구조를 키우다 (04-09)

### 뭘 했나
가장 큰 변화는 **배너 인스턴스 복제**다. 같은 롤링 배너를 2~3개 만들어서 각각 다른 설정을 줄 수 있게 됐다.

이전에는 `enabledBanners: { rolling: true, bm: true }` 같은 단순 플래그였는데, 이제 배열로 바뀌었다:
```
instances: [
  { id: 'rolling-0', type: 'rolling', settings: {...} },
  { id: 'rolling-1', type: 'rolling', settings: {...} },  ← 복제본
  { id: 'bm-0', type: 'bm', settings: {...} },
]
```

가을 배경(낙엽, 단풍, 열매)도 이때 들어갔고, 에셋을 카테고리별로 정리하는 기능도 추가됐다.

### 왜 이렇게 했을까
실무에서 같은 배너를 여러 버전으로 만드는 경우가 많다. "이 텍스트로 한번, 저 텍스트로 한번" 비교하려면 복제가 필요하다.

### 아키텍처 판단
`buildStateForInstance()`라는 함수를 만들어서, 기존 배너 템플릿 코드는 전혀 안 건드리고 인스턴스 시스템을 얹었다. 기존 코드가 `state.rolling`을 읽는 구조를 유지하면서, 렌더링 직전에 인스턴스 설정을 state에 주입하는 방식이다. 깔끔한 결정.

## v5.0 — 정리하다 (04-09)

### 뭘 했나
에셋 파일 이름을 전부 한글에서 영어로 바꿨다. `꽃-01.png` → `flower-01.png`, `겨울-팝업.png` → `winter-01-popup.png`. 879개 파일을 일괄 전환했다.

시즌별 배경 색상도 대폭 추가했다. 겨울에 은색/눈 톤, 크리스마스에 레드/그린/골드 등.

100원/1000원/5000원 같은 저가 소구점 에셋도 새로 들어갔다.

### 왜 이렇게 했을까
한글 파일명은 URL 인코딩 문제가 생기고, 코드에서 다루기 불편하다. 영어로 통일하면 동적 로딩 시스템과도 잘 맞는다.

### 자동화
`rename-assets.py` 스크립트를 만들어서 일괄 리네임했다. 하지만 새 에셋이 추가될 때 자동으로 영어 이름을 붙여주는 기능은 아직 없다. 반쪽 자동화.

## v5.1 — 이미지형을 살리다 (04-10, 이번 세션)

### 뭘 했나
이미지형 배너가 사실상 "검정 배경 + 사진"이었는데, 이번에 대폭 개선했다.

1. **이미지에서 배경색 자동 추출** — 밝은 사진이면 밝은 색, 어두운 사진이면 어두운 색으로 배경이 깔린다
2. **6색 팔레트 칩** — 사진에서 대표색 6개를 뽑아서 칩으로 보여준다. 클릭하면 배경/그라데이션이 바뀐다
3. **텍스트 자동 대비** — 배경이 밝으면 같은 색조의 어두운 톤으로 텍스트 (분홍 배경 → 진한 분홍 텍스트)
4. **블러 효과** — 이미지가 배경으로 녹아드는 경계에 블러를 넣어서 자연스럽게
5. **그래픽 에셋 지원** — 이미지형에서도 왼쪽에 3D 에셋을 넣을 수 있게

### 실수한 것들
- HSL 색상 변환을 직접 만들었다가 분홍→녹색 버그 발생. 단순한 RGB 스케일링으로 교체
- 롤링/BM만 수정하고 팝업/콘텐츠는 빠뜨림. 사용자가 발견
- 에셋 커버리지 분석에서 동적 로딩 에셋을 빠뜨려서 "0개"라고 잘못 분석

## 전체를 돌아보며

### 개발 속도
5일 동안 v2→v5.1까지 나왔다. 커밋 4개 + 미커밋 1개. 기존 프로세스였으면 이 중 하나도 제대로 못 끝냈을 분량이다.

### 반복되는 패턴
매 버전마다 같은 사이클이 반복된다:
1. 만들기 → 2. 써보기 → 3. 불편한 점 발견 → 4. 고치기 → 5. 새로운 불편 발견

이게 대화형 개발의 핵심이다. 기획서 없이, 만들면서 발견하고, 발견하면서 고친다.

### 기술 부채 누적
빠르게 만든 대가로 정리 안 된 것들이 있다:
- `banner.ts`가 900줄 넘음 (분리 필요)
- 색상 추출 알고리즘이 매직넘버로 가득 (밝기 임계값 145 등)
- 테스트 코드 0개
- `GradientPreset` 등 안 쓰는 코드 잔존

### 가장 중요한 교훈
> AI가 "할 수 있다"고 해서 다 맞는 건 아니다. 결과물을 보는 눈은 사람의 몫이다.

에셋 분석이 틀렸을 때, 색상이 이상하게 나왔을 때, 적용 범위가 빠졌을 때 — 전부 사용자가 발견했다. AI는 빠르게 만들어주지만, **판단은 내가 해야 한다**.
