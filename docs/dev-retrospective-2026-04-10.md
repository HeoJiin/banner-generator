# 배너 생성기 개발 회고 (2026-04-10 세션)

## 이 세션에서 뭘 했나?

피그마에서 콘페티 색상 바꾸는 것부터 시작해서, 이미지형 배너를 대폭 개선했다.

만든 기능들:
- 이미지 왼쪽 모서리 보더 문제 해결 (그라데이션 2px 확장)
- 이미지에서 자동으로 배경색 추출 (밝은 사진이면 밝은 색, 어두운 사진이면 어두운 색)
- 이미지에서 6색 팔레트 뽑아서 칩으로 보여주기
- 칩 선택하면 배경 + 그라데이션 + 텍스트 색상까지 한번에 바뀜
- 롤링/BM 배너에 블러 효과로 자연스러운 이미지 페이드
- 이미지형에서도 그래픽 에셋 넣을 수 있게
- 세부설정 정리 (이미지형에서 안 쓰는 그래픽 옵션 제거)
- 배경 꾸밈 드롭다운에서 '없음' 제거
- 기존 코드 버그 2개 수정 (`BG_DECORATION_ONLY_ASSETS` 미정의, `hundred` 중복 키)

---

## 어떤 식으로 개발했나?

### 한마디로: "만들면서 고치기"

일반적인 개발 프로세스는 이렇다:
> 기획서 작성 → 디자인 → 개발 요청 → 개발 → QA → 배포

이 세션에서는 이렇게 했다:
> 화면 보면서 "이거 좀 이상한데" → Claude한테 말하기 → 30초 후 수정됨 → "오 근데 이것도" → 반복

기획서가 없다. 화면을 보면서 필요한 걸 그때그때 요청했다. 하나 고치면 다음에 고칠 게 눈에 보이고, 그걸 또 고치면 또 보이는 식이다.

**예시 흐름:**
1. "이미지 배경이 항상 검정이네" → 이미지에서 색상 추출하자
2. 추출은 되는데 하나만 나오네 → 6개 뽑아서 고를 수 있게 하자
3. 색상은 바뀌는데 텍스트가 안 보이네 → 텍스트도 자동 대비시키자
4. 근데 분홍 배경에 녹색 텍스트가 나오네 → 알고리즘이 잘못됐구나, 고치자

이런 식으로 하나의 기능이 꼬리에 꼬리를 물고 발전했다.

---

## 잘한 것

### 1. 피드백이 빠르다
아이디어가 나오면 30분 안에 동작하는 코드가 나온다. 기존이라면 "이런 기능 필요해요" 티켓 쓰고 2주 기다려야 했을 일이다.

### 2. 디자이너의 감각이 바로 반영된다
"분홍 배경이면 어두운 분홍 톤으로 텍스트" 같은 건, 기획서로 쓰면 "배경색의 HSL 값에서 L을 25%로..." 이런 식이 된다. 근데 대화로 하면 "알지?" 한마디로 끝난다. AI가 알아서 번역한다.

### 3. 기존 코드를 잘 재활용했다
그래픽형에서 쓰던 에셋 그리드(`CategorizedAssetGrid`)를 이미지형에도 그대로 가져다 썼다. 색상 추출 유틸도 기존 `colorExtract.ts`에 함수만 추가하는 식으로 확장했다.

## 실수한 것 (오답노트)

### 실수 1: AI 분석을 그냥 믿을 뻔했다

에셋이 키워드별로 몇 개 있는지 분석해달라고 했더니, "info는 0개, growth는 0개"라고 했다. 코드에서 빈 배열(`[]`)만 보고 판단한 거였다.

실제로는 파일시스템에서 `info-magnifier-1.png` 같은 파일을 실시간으로 읽어오는 구조였다. 코드만 보면 빈 배열인데, 실행하면 16개가 로딩되는 것이다.

**내가 "어딜 보고 분석한거야?"라고 안 물었으면** 틀린 데이터로 다음 작업을 진행할 뻔했다.

> 교훈: AI가 분석해준 걸 한번 더 확인하는 습관이 필요하다. 특히 숫자가 나오면.

### 실수 2: 색상 변환을 직접 만들었다가 버그

밝은 분홍 배경에 어두운 분홍 텍스트를 넣으려고, RGB를 HSL로 바꿨다가 다시 RGB로 바꾸는 코드를 AI가 직접 짰다. 그랬더니 분홍 배경에 녹색 텍스트가 나왔다.

결국 복잡한 HSL 변환을 버리고, 그냥 RGB 값을 동일 비율로 어둡게 만드는 단순한 방식으로 바꿨다. `rgb(255, 200, 220)` → 비율 유지하면서 어둡게 → `rgb(70, 55, 60)`. 같은 분홍톤이 나온다.

> 교훈: 복잡하게 풀 수 있다고 복잡하게 푸는 게 아니다. 단순한 방법이 버그도 적다.

### 실수 3: 절반만 고치고 넘어감

배경색을 이미지에서 추출한 색으로 바꿀 때, 롤링/BM 배너만 바꾸고 팝업/콘텐츠 배너는 빼먹었다. 4개 배너 타입이 같은 패턴인데 2개만 고친 것이다.

사용자가 "팝업이랑 콘텐츠는 적용 안 됐어"라고 말해줘서 알았다.

> 교훈: 같은 패턴이 여러 군데 있으면, 한번에 전부 찾아서 고쳐야 한다.

### 실수 4: 기존 버그를 모르고 있었다

코드에 `BG_DECORATION_ONLY_ASSETS`라는 변수를 쓰는데 정의가 없었다. 실행하면 에러가 터지는 상태. 또 `hundred`라는 키가 두 번 정의되어 있어서 타입스크립트 에러가 나고 있었다.

이건 이 세션에서 만든 버그가 아니라 이전부터 있던 건데, 작업 시작할 때 체크를 안 했다.

> 교훈: 작업 시작 전에 기존 에러가 있는지 한번 확인하자.

## 이 방식의 장단점

### 장점
| 항목 | 설명 |
|------|------|
| **속도** | 기능 10개를 한 세션에 만들었다 |
| **정확도** | 실물을 보면서 판단하니까 "이게 아닌데" 하는 게 빨리 보인다 |
| **유연함** | 기획서에 없는 것도 즉석에서 추가할 수 있다 |
| **저비용** | 디자이너 1명 + AI로 프로토타입 수준이 아닌 실동작 앱이 나온다 |

### 단점
| 항목 | 설명 |
|------|------|
| **검증 부족** | 빨리 만드느라 "이게 진짜 맞나?" 확인을 건너뛴다 |
| **전체 그림 없음** | 기획서가 없으니 어디까지 만들지 기준이 애매하다 |
| **기술 부채** | 돌아가긴 하는데, 나중에 문제될 수 있는 코드가 쌓인다 |
| **AI 실수 위험** | 분석이 틀리거나 알고리즘에 버그가 있을 수 있다 |

## 다음에 더 잘하려면

### 1. 작업 전 기존 상태 체크하기
타입스크립트 에러가 있는지, 깨진 기능이 있는지 먼저 확인하고 시작하자.

### 2. AI 분석 결과는 한번 더 확인하기
숫자, 목록, "없다/있다" 판단은 꼭 실제로 맞는지 크로스체크하자. 특히 동적으로 로딩되는 데이터는 코드만 봐서는 모른다.

### 3. 같은 패턴은 한번에 전부 고치기
"4개 배너 타입 중 2개만 고치고 넘어가지 말자." 검색해서 같은 패턴 전부 찾고, 한번에 수정하자.

### 4. 복잡한 알고리즘은 단순하게 먼저
HSL 변환처럼 수학적으로 정확한 방법보다, RGB 스케일링처럼 단순하지만 충분히 좋은 방법을 먼저 시도하자.

### 5. 한 기능 끝나면 잠깐 멈추기
기능 하나 만들면 바로 다음으로 넘어가지 말고, 2~3개 다른 케이스로 테스트해보자. "분홍 이미지에서도 되나? 어두운 이미지에서도 되나? 무채색이면?"

## 브런치 글 요약: 디자이너가 Claude Code로 개발하는 법

### 내가 한 건 코딩이 아니라 "판단"이다

코드를 한 줄도 직접 안 썼다. 내가 한 건:
- 화면을 보고 "이게 이상하다"고 말하기
- "이런 식으로 되면 좋겠다"고 방향 잡기
- 결과를 보고 "맞다/아니다" 판단하기
- AI가 틀렸을 때 "그거 아닌데?" 잡아주기

### 개발 지식이 필요한가?
코드를 읽을 줄 몰라도 된다. 대신 **결과물을 보는 눈**이 있어야 한다. "이 색상 조합이 자연스러운가", "이 UI가 사용자에게 혼란을 주진 않는가" 같은 판단은 AI가 못한다. 그게 내 역할이다.

### 주의할 점
AI가 "할 수 있다"고 해서 다 맞는 건 아니다. 이 세션에서도 에셋 분석이 틀렸고, 색상 알고리즘에 버그가 있었다. **AI를 시니어 개발자가 아니라, 실력 좋지만 가끔 실수하는 주니어 개발자**로 생각하면 딱 맞다. 결과물은 내가 검수해야 한다.
