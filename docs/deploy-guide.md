# 배포 가이드 (비개발자용)

> 배너 생성기와 관리자 API를 수정하고 배포하는 전체 과정을 설명합니다.

---

## 전체 흐름 요약

```
코드 수정 → 로컬에서 확인 → 커밋(저장) → 배포
```

쉽게 말하면:
1. 파일을 수정한다
2. 내 컴퓨터에서 잘 되는지 확인한다
3. 수정 내역을 기록(커밋)한다
4. 서버에 올린다(배포)

---

## 용어 설명

| 용어 | 쉬운 설명 |
|------|-----------|
| **Git** | 파일 수정 이력을 관리하는 도구. "저장 히스토리"라고 생각하면 됨 |
| **커밋 (Commit)** | 수정 내역을 하나의 저장 포인트로 기록하는 것. 게임의 "세이브" 같은 개념 |
| **브랜치 (Branch)** | 원본을 건드리지 않고 별도 공간에서 작업하는 것. 복사본에서 작업 후 합치는 방식 |
| **머지 (Merge)** | 브랜치에서 작업한 내용을 원본(main)에 합치는 것 |
| **main** | 원본 브랜치. 실제 배포되는 코드가 여기에 있음 |
| **Railway** | 우리가 쓰는 서버 호스팅 서비스. 코드를 올리면 자동으로 서버에서 실행해줌 |
| **`railway up`** | 내 컴퓨터의 코드를 Railway 서버에 올리는 명령어 |

---

## 우리 서비스 구성

| 서비스 | 설명 | 주소 |
|--------|------|------|
| **배너 생성기** | 배너 만드는 웹 도구 | miraculous-empathy-production.up.railway.app |
| **관리자 API** | 다운로드 로그 조회 + 인사이트 대시보드 | (별도 주소) |

두 서비스는 별도로 배포합니다. 하나를 수정해도 다른 하나에 영향 없음.

---

## 배포 과정 (오늘 한 것 기준)

### 1단계: 현재 상태 저장 (커밋)

수정하기 전에 지금 상태를 먼저 저장해둡니다. 나중에 문제가 생기면 이 시점으로 돌아올 수 있습니다.

```
git add [수정한 파일들]
git commit -m "어떤 변경인지 설명"
```

### 2단계: 새 기능은 브랜치에서 작업

원본(main)을 바로 수정하면 위험하므로, 복사본(브랜치)을 만들어서 거기서 작업합니다.

```
git checkout -b feat/기능이름    ← 새 브랜치 생성
(코드 수정)
git commit -m "기능 설명"        ← 수정 내역 저장
```

### 3단계: 로컬에서 확인

내 컴퓨터에서 서버를 띄워서 잘 동작하는지 확인합니다.

- 배너 생성기: `npm run dev` → http://localhost:3000 접속
- 관리자 API: `npm start` → http://localhost:3100 접속

### 4단계: 원본에 합치기 (머지)

확인 끝나면 브랜치 작업물을 원본(main)에 합칩니다.

```
git checkout main              ← 원본으로 이동
git merge feat/기능이름         ← 브랜치 내용 합치기
```

### 5단계: 배포

Railway 서버에 올립니다.

**배너 생성기 배포:**
```
cd banner-generator
railway service miraculous-empathy
railway up
```

**관리자 API 배포:**
```
cd banner-admin-api
railway service banner-admin-api
railway up
```

`railway up`을 치면 코드가 서버에 업로드되고, 자동으로 빌드 → 실행됩니다. 1~2분 정도 걸립니다.

## 오늘 배포한 기능들

### 배너 생성기 — 메인 그래픽 미세 조정
- 팝업/콘텐츠 배너에서 3D 에셋의 크기·위치를 슬라이더로 조정 가능
- 크기: 85%~100%, 좌우: ±20px, 하단 이동: 팝업 0~20px / 콘텐츠 0~80px
- 초기화 버튼으로 원래대로 복원 가능

### 관리자 API — 로그 삭제 기능
- 로그 상세 화면에서 삭제 버튼 추가
- 확인 팝업으로 실수 방지
- 테스트 데이터를 지워서 인사이트 정확도 유지

## 자주 묻는 질문

**Q: 배포하면 기존 데이터 날아가나요?**
A: 아닙니다. 코드만 교체되고, 데이터베이스(로그, 썸네일 등)는 그대로 유지됩니다.

**Q: 배포 후 문제가 생기면?**
A: Git에 이전 커밋이 저장되어 있으므로, 이전 버전으로 돌아가서 다시 배포할 수 있습니다.

**Q: 두 서비스를 동시에 배포해야 하나요?**
A: 아닙니다. 독립적이므로 수정한 쪽만 배포하면 됩니다.
